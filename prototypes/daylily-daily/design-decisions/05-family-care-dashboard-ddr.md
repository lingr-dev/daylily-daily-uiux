# Design Decisions Record (DDR) - 萱草日签：家人照护（异常处理大盘）

本文档记录了 `05-family-exceptions.html`（家人照护控制台）以及相对应的首页精简逻辑的设计共识。这是整个产品“家庭督促”社交闭环的心脏。

## 1. 架构奥卡姆剃刀：精简一切交叉路径
**[决策背景]**
由于首页既有“详情”入口，又有顶部的“家人照护”切换口，很容易让长辈对于同一个目标产生路径迷失。

**[设计共识]**
- **砍掉前台历史流水账**：将常规的“他早晨吃了、中午吃了”的打卡流水账历史，剥离出核心监护页，下放归档至地步导航的**【日历】**模块，作为低频查询备用。
- **首页时间轴瘦身**：首页 (01) 下方的时间轴明确定义为“今日高优动态 TOP 3”。**彻底移除了“详情 >”跳转入口**。
- **确立唯一核心域**：任何涉及家人大盘总览的诉求，均由顶部显眼的【家人照护】胶囊按钮唯一承接，降低认知负担。

---

## 2. 拥抱极端纯粹的异常态 (Exception-Driven UX)
**[决策背景]**
如果把“没吃药”的红点混在满屏“已吃药”的绿点中，长辈很难一眼抓住重点。监护视角的诉求是“谁需要我催？”，而不是“谁表现良好”。

**[设计共识]**
- **干掉成功项**：在“家人照护”面板中，**绝对不展示**已经按时完成的用药任务。
- **放大警报势能**：页面使用高压驱动的设计，任何长辈或由于疏忽延误吃药的任务，将直接膨胀为一张张巨大的**红色警示态漂浮卡片**（红底、频闪边框）。
- **社交督促的终点**：在每张异常卡片下方，设置硕大的绿色主按钮**【微信拍一拍提醒】**或**【发送微信通知】**。点击后即刻唤起微信系统的服务通知下发能力，完成从“发现异常 -> 远程催开药”的社交连招闭环。

---

## 3. “岁月静好”空状态（情绪价值供给）
**[决策背景]**
医疗健康类产品往往充满冰冷的数据。当家中老人都按时吃了药时，系统如果只显示白板，这是极其浪费产品情绪价值的。

**[设计共识]**
- **建立安全感奖励**：当“家人照护”大盘中没有任何漏服警告时（即所谓 Empty State），用页面最中心的巨大绿色盾牌及柔和的心跳光晕包裹视线。
- **疗愈向文案**：配注“今日家人用药均已达标，岁月静好”文案。赋予使用者（通常是焦虑的晚辈或伴侣）极大的心理安慰和精神释重感（Peace of mind）。以此正向激励用户每天坚持留存查看。

---

## 4. 底部路由时代的顶层视图切换架构 (View Switching Architecture)
**[决策背景]**
由于已引入底部的全局原生 Tabbar 进行页面级路由（首页、日历、我的），如果顶部“我的用药/家人照护”依然使用页面级跳转，会导致 Tabbar 重载闪烁，破坏小程序原生感。且 WeUI 官方扩展库推荐的 `mp-tabs`（全宽下划线选项卡）视觉上过于厚重、古板，与本品要求的“灵动、无攻击性”理念相悖。

**[设计共识]**
- **摒弃传统路由与默认 Tab 样式**：坚决不分离成两个物理页面，同时不套用 WeUI 的标准 `Tabs` 样式。继续采用基于 WeUI `flex` 基础规范手写的“灵动岛式”圆润胶囊。
- **自定义胶囊 + Swiper 状态同构 (The Engine)**：视图底层切换规范对齐微信官方的最佳实践思路，利用底部的原生 `<swiper>` 组件来物理承载“打卡主屏”与“家人大盘”两个并列视图，避免 DOM 强刷。
- **双向滑动闭环**：通过机制绑定，确保点击顶部胶囊可使底盘视图滑动；同时用户手指在页面上左右滑动页面时（触发 bindchange），顶部胶囊的高亮色块以及常驻红点亦能丝滑联动跟随。

## 5. 干预升级逻辑 (Intervention Escalation Logic)
**[决策背景]**
在家人照护场景下，不同的漏服事件其临床风险和社交语境是完全不同的。如果对于轻量保健品延时和高危处方药延时都使用同一种强度的“提醒”模板，要么会引起被提醒人（尤其是年轻晚辈）的不满与社交压力，要么会贻误长辈的重要病情。设计不能仅仅为了视觉统一而放弃对业务上下文的感知。

**[设计共识]**
- **上下文感知动作 (Context-Aware Actions)**：异常卡片底部的操作按钮不再是静态、一刀切的，而是必须根据“药品危险等级 + 延时时长 + 被提醒人身份”进行动态的降级或升级。
- **强干预 / 高危场景**：对于长延时（如 >2 小时）或高危处方药（如降压药）漏服，尤其是针对长辈。必须强制升级为最高干预级别。UI 表现为带渐变强底色的 **【微信语音通话】** 按钮。摒弃可能被忽略的文字消息，直接以最高通讯优先级介入。
- **弱干预 / 轻量场景**：对于短延时（如 <1 小时）或保健品（如维生素）漏服，尤其是针对年轻晚辈。降级为轻社交互动。UI 表现为留白更多的线框按钮 **【微信拍一拍提醒】**。提供一种不留痕迹、无心理负担的“戳一下”关怀，克制产品的打扰感。

## 6. 异常警报闭环机制 (Alert Resolution Mechanism)
**[决策背景]**
原型的早期版本在异常卡片右下角仅提供了一个“✖️（忽略）”按钮。但这在产品逻辑上存在语义模糊和安全隐患：消除卡片是因为“长辈已经补服了”还是仅仅“我不想看了”？此外，如果晚辈确认老人已服药，缺乏代为打卡的快速入口，会导致警报无法形成数据闭环。

**[设计共识]**
- **取消单向忽略**：将 `✖️` 替换为语义更宽泛的 `···`（更多/处置警报）按钮。
- **引入 Action Sheet 强制归因**：点击处置后，不直接让卡片消失，而是从底部唤起处置面板，提供三个明确的收口选项：
  1. **【已确认家人服药】**：代长辈完成打卡，补齐服药记录，同时消除警报（推荐的正向闭环）。
  2. **【稍后再次提醒】**：知情并暂缓（Intentional Deferral），比如长辈正在外出，设定 30 分钟后再重新警报。
  3. **【已知晓，暂不干预】**：放弃本次干预（假阳性或无需干预），单次忽略并将警报归档。
- **情绪出口**：只有提供了清晰的收口机制，大盘才能真正被清空，用户才能体验到“岁月静好”的空状态带来的情绪奖励。

## 7. 跨组聚合归属展示 (Cross-Group Attribution Slot)
**[决策背景]**
照护者可能同时是多个家庭组的成员（接受多份邀请，或 Pro 用户后续可创建多个家庭组）。同一面板里来自不同家庭组的异常卡片若不区分归属，会出现"同名家人来自不同组"的混淆；早期实现将组名与成员名拼成 `{group} • {name}` 单行展示，在窄屏与长组名场景下极易换行，破坏视觉节奏。

**[设计共识]**
- **独立 kicker 槽位**：在成员主名（`text-[15px] font-bold`）上方放一行小号弱灰文本（`text-[10px] font-semibold text-neutral-400 leading-tight tracking-wide`），承载家庭组归属，独立于主名行，杜绝挤压换行。
- **超长截断**：`truncate max-w-[140px]`，超出以省略号收口；不允许换行。
- **何时渲染**：
  - **非默认家庭** → 始终渲染该组的 `groupName`。
  - **默认家庭 + 受邀者视角** → 命名层 `translateGroupDisplayName` 翻译为"{owner}的家庭"（DDR 11 §3.2 ③），渲染翻译后的字符串。
  - **默认家庭 + owner 视角** → 仍渲染原始 `groupName`（不再像早期实现那样空着）；为后续 Pro 多组场景预留视觉一致性，避免"用户从单组升级到多组时卡片信息密度突变"。
  - **自身事件**（`alert.subjectId === viewer.defaultSubject.id`） → kicker 不渲染（DOM 不占位），主名直接显示"我"，避免冗余。
- **命名规则统一引用**：成员主名与组名的全部回退分支都走 [`design-specs/20-runtimes/miniprogram/display-name-layer.md`](../../../../design-specs/20-runtimes/miniprogram/display-name-layer.md) 中定义的 `resolveSubjectDisplayName` / `translateGroupDisplayName`，本 DDR 不重复回退树定义；规则源在 DDR 11 §3 与 DS-40-authz-001 §5。
