# DDR 11: 数据权限边界与多组命名策略 (Permissions & Naming Strategy)

## 1. 决策背景
随着家庭组模型（Family Group）复杂度的提升，暴露出了跨组数据权限（如“微信成员代管权”）以及显示侧的灾难级体验缺陷（全站显示长串无意义 ID、多个“我的家庭”重名导致认知崩溃）。本 DDR 旨在固化针对这三大核心痛点的逻辑解法，指导前端渲染和接口组装。

## 2. 数据权限与组员代管边界

本业务的权限隔离机制不再单纯依赖“有无微信”，而是基于 **“所属组角色（Owner vs Member）”** 进行高维拦截。

### 2.1 成员代管权
- **Owner 的统管权**：在“我 Own 的家庭组”内，Group Owner 具有最高代管权限，可为任何人（影子账户、微信账户）创建和修改用药计划。这覆盖了“监护人照护有微信的孩子/老人”的核心场景。
- **Member 的受限权**：对于受邀加入组的 Group Member，严防跨级越权。只能为组内的“影子账户”代管用药；对于该组内其他的微信成员，置灰其选择入口（“仅限管理员或本人操作”）。
- **执行者的最终决定权**：如果 Owner 替组内某微信成员（如 10 岁的儿子）创建了计划，该微信成员本人始终拥有编辑、停用该计划的最高优先权。

### 2.2 跨组数据聚合
- 首页「家人照护」告警列表、日历视图等跨组成员展示入口，必须全量聚合当前用户所参与的 **所有组（Own & Joined）** 中的非私密动态。

## 3. 命名策略 (Naming & Display Strategy)

在当前“Group 为单一全局记录（免费版不可改名）”且不支持“成员端存 Group 自定义别名”的架构下，采用 **“纯动态视图翻译 (Dynamic Translation)”** 以及 **“脱敏短 ID 兜底”** 是唯一解。

### 3.1 用户本身的 Public Name 回退树
- **① 本尊固定翻译**：只要列表数据的 `user_id` 等于当前登录用户，前端强制渲染为 **“我”**。
- **② 微信昵称 (Public Name)**：用户主动授权或主动补填的称呼。
- **③ 短 ID 兜底**：若用户拒绝授权且未补填称呼，强制回退为 **`用户_A1B2`**（取 UUID 短位），严禁长串 UUID 破坏卡片 UI 布局。

### 3.2 解决群组“撞名”的动态翻译
免费版由于无法更改默认组名 `我的家庭`，直接透传会导致列表中出现多个相同名字的群。

- **动态判断**：BFF 或后端在输出 Family Group List 时，判断该组是否为系统默认名（`is_default_name == true` 或 `group_name === "我的家庭"`）。
- **Owner 视角**：若当前请求者是群主，原样下发 `我的家庭`。
- **Guest 视角**：若请求者是非群主的受邀成员，**内存层直接替换组名为 `[Owner_Public_Name]的家庭`**（如 `建国的家庭` 或 `用户_A1B2的家庭`）。
- 该策略巧妙绕过了改名限制，彻底解决受邀者的“找群”噩梦。

### 3.3 邀请链路的轻量兜底
- **拦截器（非强制）**：当处于纯匿名短 ID 状态（`用户_A1B2`）的用户点击“生成邀请链接”时，应弹出轻量提示请求补填“家中称呼”（如：爸爸）。若跳过，卡片即原样输出短 ID 邀请。

## 4. UI 呈现规范

| 页面组件 | 规范 |
|----------|------|
| **首页：异常卡片** | 采用 `[翻译后的组名] • [家人 Public Name]` 模式（如：`老李的家庭 • 用户_A1B2 漏服了`），明确上下文关系。 |
| **Profile (“我的” 页)** | 顶部必须 **透传底层真实 Public Name**（哪怕是 `用户_A1B2`），并附加显眼编辑按钮（✏️）。这是用户自证“我在别人眼里叫什么”的唯一视觉锚点。严禁使用“萱草新朋友”等假数据兜底。 |
| **日历视图：防爆折叠** | 若跨组成员总数 > 5 人，日历流的当日事件默认进入“折叠聚合模式”；仅展开有异常告警的成员，按时完成的成员合并收起。 |
