## 1. 页面目标与用户任务

- **本页要完成的任务**：作为小程序主入口，在「我优先」前提下完成当日个人用药打卡（或与**订阅消息深链**对齐的 occurrence）、展示家人类异常与健康动态摘要；通过顶部胶囊在**同页**切换「我的用药 / 家人照护」两种模式；**仅为纯新用户（H-ONBOARD）** 在首页提供直达创建计划的引导；**已登录用户为自己管理用药计划统一走「我的」页**（见下条）。
- **为自己管理用药计划（V1 统一入口）**：除 **H-ONBOARD** 冷启动外，**为自己创建、查看、编辑**用药计划 **一律**经底部 Tab **「我的」**（`02-profile.html`）→ **「日常用药计划」**区块（原型为 **「我的用药」** 卡片）进入，再衔接 `03-plan-list.html`、`04-plan-create.html` 等（以 Profile 信息架构为准）。首页 Hero **Idle** 不提供跳转。**后续**多途径交互再单独立项改版。
- **主用户角色**：已登录微信用户；细分见 `role-permission-matrix.md`（被照护者、照护者、单机用户、纯照护者边界）。
- **名词：Deep link（深链）**：指用户点击微信 **订阅消息 / 服务通知** 卡片打开小程序时，启动路径上携带的 `query` 参数（如 `plan_id`、`occurrence_id`）。承接逻辑见 `req-specs/.../flow-reminder-entry-punch.md`。并非独立页面，只是**带参进入当前小程序路由**的一种方式。
- **入口 / 出口**：
  - **入口**：Tab「首页」、订阅消息深链（参数见上）、自计划创建流程完成后的返回（若有）。
  - **出口**：Hero 仅在 **Ready / Warning** 下用于 **提交打卡**；**Idle / Waiting / All Done** 下无 Hero 导航；**H-ONBOARD** 新手 CTA → `04-plan-create.html` / `?role=family_new`；底部 Tab → `06-calendar.html`、`02-profile.html`（**为自己管计划的统一入口**）；健康动态卡片 → （工程定详情路由）。**胶囊切换不改变路由、不 `navigateTo` 新页。**

## 2. 展示数据范围

### 2.1 数据主体与来源

| 区块 | 数据主体 | 读模型说明 |
|------|----------|------------|
| Hero + 「接下来」 | 仅 **「我的用药」模式** 下展示。当前登录用户作为 **执行者** 的下一待处理 **当日** 用药 occurrence | 仅当 **当日内存在至少一条排程** 时进入 Waiting/Ready/Warning/Missed 行为链；深链带 `plan_id` / `occurrence_id` 时优先聚焦该条（见 `flow-reminder-entry-punch.md`） |
| 进度环 | **仅统计当日内、由计划产生的排程** | **分母** = 当日内有排程的 occurrence 总数（**当日无任何排程则分母为 0，不纳入计算**）。**分子** = 其中已完成打卡的条数。**不设**撤销回退、不做跨日合并、不做多计划加权 special case。展示形态：`Idle` 态时进度环为 **idle 视觉**（空槽 / 与原型 `idle` 态 ring 一致，纯照护者 H-IDLE-CAREGIVER 亦然） |
| 灵动岛红点 | 家人维度是否存在待照护者感知的 **异常摘要**（如漏服） | 与 `design-decisions/01-uiux-consensus-ddr.md`「静默提醒」一致 |
| 「我的用药」— 今日健康动态 | 「我的用药」模式主内容区之一：混合信息流 | **固定 Top 3**；**按排程事件发生时间倒序**（最近发生的在前）。**不提供「查看更多」**（与顶部胶囊切换信息架构重复）。**私密计划**对非本人不可见元数据（`R-AP-3`、MJ2） |
| 「家人照护」— 主列表 | **「家人照护」模式**独占主区（见 3.2） | **不复用**「我的用药」的 Hero 打卡布局。内容为 **异常态 / 待监督的排程事件** 列表，语义上是「今日健康动态」所代表数据类型的 **完整展开版**（仍以卡片列表呈现）。条数与排序规则与上：**Top 3**、**事件时间倒序**、无「查看更多」 |

### 2.2 过滤、排序与时间窗

| 项目 | 规则 |
|------|------|
| 「我的用药」数据主体 | 当前用户本人为执行者的计划所产生的 **当日** occurrence；家人动态来自照护组可见成员（排除私密计划） |
| 「家人照护」数据主体 | 需监督视角下的 **异常态 / 待处理排程事件**（非简单复用「我的用药」列表数据源） |
| 健康动态 / 家人列表排序 | **排程事件发生时间倒序**，最近在前；**同序内**卡片样式（异常描边、灰阶弱化等）仍可遵循 `01-uiux-consensus-ddr` 3.4，与排序不矛盾 |
| 条数上限 | **两处列表均为 Top 3**（「我的用药」摘要区、「家人照护」主列表）；超出部分不展示、不提供展开入口（由胶囊切换承担导航心智） |
| 时间窗 / 时区 | Hero 状态机以 **计划当地时间** 与用户设备展示一致为前提；**服务端权威时间**用于判定是否超时（与 DDR 时间窗一致） |
| occurrence 切换 | 当某次服药 **已错过**（DDR：计划后大于 1 小时）：Hero **不长期停留漏服主态**，推进到 **下一当日 occurrence**；若无下一则跌入 **Idle（今日无后续排程）** 或 **All Done**（见 3.1） |

### 2.3 三种主态（与原型视图切换）

| 主态 ID | 条件（产品口径） | 主内容区 |
|---------|------------------|----------|
| H-ONBOARD | 全局 **个人计划数 = 0** 且 **家庭计划数 = 0**（纯净新用户，`Total Plans == 0`） | 仅展示新手引导：`home-empty-view`；隐藏顶部「我的用药 / 家人照护」切换器 |
| H-ACTIVE | **个人计划数 > 0** | 「我的用药」默认：`home-active-view`，含 **Hero + 今日健康动态** 与胶囊；若切至「家人照护」则主区切换为家人列表（无 Hero），**路由不变** |
| H-IDLE-CAREGIVER | **个人计划数 = 0** 且 **家庭计划数 > 0**（纯照护者） | `home-active-view`：**Hero 固定 Idle**（琥珀、进度环 idle 视觉）；文案侧可弱提示「在「我的」— 日常用药计划中添加自己的用药」；「家人照护」下仍只展示家人异常列表 |

原型调试：`toggleViewMode()` 仅在本地模拟 H-ONBOARD vs 非引导；**工程不得以调试默认值覆盖 H-ONBOARD 数据规则**。

### 2.4 空数据与无权限

| 场景 | 区分方式 | 界面表现 |
|------|----------|----------|
| 新用户无计划 | H-ONBOARD | 欢迎文案 +「立即创建用药计划」+「我是照护者，为家人创建」 |
| 有个人计划但 **当日内无任何排程** | 当日 occurrence 集合为空 | Hero 进入 **Idle**（非 All Done）；副文案建议区分于纯照护者：例如「今日暂无用药安排」；若需查看或调整计划 → **「我的」— 日常用药计划**。进度环 **idle 视觉**，分母视为 0 |
| 私密计划 | 接口层对非本人返回不可见 | 任何列表 **不包含** 可识别元数据 |

## 3. 控件状态与控制逻辑

### 3.1 Hero 按钮（仅「我的用药」模式；时间闸状态机）

以下条件仅当 **当日内排程分母 > 0**、且当前 Hero **不处于任一 Idle 子态**（纯照护者 Idle / 今日无排程 Idle）时，进入 **Waiting → Ready → Warning → Missed（行为）** 链；否则见表末 **Idle / All Done**。

条件中的「计划时间」指 **当前聚焦 occurrence 的计划服药时点**。**±1 小时** 阈值取自 DDR（`01-uiux-consensus-ddr.md` 第 3.1 节）；若 req-specs 日后收口不同数值，以 **req-specs** 为准并改 DDR。

| 状态 ID | 显示条件（摘要） | 主文案 / 副文案 | 颜色 / 可点 | 点击行为 |
|---------|------------------|-----------------|-------------|----------|
| Waiting | 距计划时间 **大于** 1 小时 | 「安心休息」/ 下一次用药时间 | 灰、不可点 | 无 |
| Ready | 计划前 **1 小时内**（含到点） | 「我吃过了」/ 药名·剂量 | 品牌主色、可点 | 提交 **准时打卡**（F-PUNCH）；成功后刷新分子或进入下一 occurrence |
| Warning | 计划后 **1 小时内** | 「我补吃了」/「稍晚了一点」 | 警告色、可点 | 提交 **补吃打卡** |
| Missed（行为） | 计划后 **大于** 1 小时 | 不在 Hero 长期展示漏服主态 | — | 切至 **下一当日 occurrence** 的 Waiting/Ready/…；漏服由列表承接 |
| All Done | **当日内分母 > 0** 且 **分子 = 分母**（当日排程已全部完成） | 「今日已达标」 | 成功色、不可点 | 无 |
| Idle（纯照护者） | 个人计划数 = 0 且家人计划数 > 0 | 「记录我的健康」/「先照顾好自己」 | 琥珀、**不可点** | 无；为自己建计划 **仅**经 **「我的」→ 日常用药计划（我的用药）**（见 §1 统一入口） |
| Idle（今日无排程） | 个人计划数 > 0 **且** 当日内无任一排程 occurrence | 主文案建议「今日暂无用药安排」等（与 Idle-Caregiver 区分） | 琥珀或中性、**不可点** | 无；查计划 / 新建下一周期用药 → **「我的」— 日常用药计划** |
| **进度环** | 与上表联动 | **分母 > 0**：环进度 = 分子/分母；**分母 = 0 或 Idle 态**：**idle 视觉**（与原型 `state-idle` ring 一致）。纯照护者 **H-IDLE-CAREGIVER** 下必为 idle 环 | — | — |

**不设**打卡撤销后的回滚；不跨计划做特殊加权（多条计划只合并为当日排程总数）。

**光晕动效**：可点击态（Ready、Warning）`glow-breathe`；**不可点击态**（含 Waiting、All Done、**两种 Idle**）关闭（与原型一致）。

**「家人照护」模式下**：不展示本节 Hero 整块（见 3.2）。

### 3.2 灵动岛「我的用药 / 家人照护」（仅首页单页切换）

| 控件 | 显示条件 | 行为 | 红点 |
|------|----------|------|------|
| 整块切换器 | H-ACTIVE 或 H-IDLE-CAREGIVER；**非** H-ONBOARD | **不切换路由、不开新页面**；仅在 `01-home` 内切换 **local mode**（`me` \| `family`） | 家人侧有待感知异常时显示 `#alert-dot` |
| 「我的用药」 | `mode=me` | 展示 **Hero（若适用）+ 今日健康动态（Top 3）** | — |
| 「家人照护」 | `mode=family` | **隐藏**「我的用药」的 **Hero 打卡区**；主区域仅为 **家人侧异常态排程事件** 列表（Top 3、事件时间倒序），**不复用**「我的用药」布局与数据源逻辑；语义 = 今日健康动态之完整版 | 同上 |

工程实现可为两套 DOM 互斥 `display`，或两容器切换，**不得** `wx.navigateTo` 到新页面。

### 3.3 今日健康动态（「我的用药」）与能量补充

| 行类型 | 展示条件（摘要） | 右侧标签示例 | 「能量不足 / 补充」 |
|--------|------------------|--------------|---------------------|
| 列表行（混合） | 数据源见 2.1；固定 **Top 3**、**排程事件时间倒序** | 「漏服」「下一个」等 | **仅当该条对应「当前用户为执行者」的自己的事件** 且触达失败需补授权时展示「能量不足，未发送」+「补充」。**点击「补充」**：直接调起 **微信消息订阅** 能力（如 `wx.requestSubscribeMessage`），**不做独立小程序业务页** |
| 家人等其他执行者事件 | 可出现于「我的用药」摘要或「家人照护」列表 | — | **不出现** 补充 CTA（非本人不可代调本人能量） |

首页 **不提供**「一键提醒」类额外按钮（DDR 3.4）；**补充** 为例外，且 **仅本人事件**。

> **原型修正**：`01-home.html` 中家人卡片上的「补充」仅为历史占位；按本契约应仅出现在 **本人** 相关卡片上。

## 4. 与其他页面 / 模块的依赖

- **必须已具备的前置数据**：登录态、用户照护组成员关系、计划与排程投影、打卡写接口（Ready/Warning 提交）。
- **深链承接**：`flow-reminder-entry-punch.md` — 参数缺失时退化为普通一键打卡；occurrence 过期时提示并允许 raw punch。**若用户从未创建计划却收到订阅消息**：是否可能发生、是否应先走建计划，**依微信下发策略与运营配置验证**（本契约不强制流程）。
- **共用组件**：主题色与间距 token 见 `themes/default.css` 与 DDR 第 2 节。
